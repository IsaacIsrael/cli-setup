# Triage Leftover

Run **effort-impact triage** over a milestone's [`leftover`](../../docs/CONTEXT.md) container: score every leftover item, present the classification and proposed routing, and — only on the developer's confirmation — execute it. The leaf reached by the `/triage` router when the target carries `leftover`.

The quadrant model (⚡ quick win / 🏆 real win / 🍬 nice win / 🕳️ time sink) and the routing each implies are defined in the glossary ([CONTEXT.md](../../docs/CONTEXT.md), Planning Workflow). This skill is the operational process; do not restate the definitions.

Every comment posted to the issue tracker during this skill **must** start with the triage AI disclaimer from [`triage-issue`](../triage-issue/SKILL.md):

```
> *This was generated by AI during triage.*
```

## Process

### 1. Resolve the container

Resolve the target milestone (argument, else the active milestone inferred from the current issue/branch per [branch-issue-resolution.md](../../docs/branch-issue-resolution.md)) and its `leftover` container. Read every unchecked leftover item.

Done when you have the full list of open items for one milestone.

### 2. Score each item

For each item, judge **effort** and **impact** and assign a quadrant. The boundaries:

- **Effort** — a soft ~1h anchor: **low** = aggregatable (fits ~1h, can join the refinement issue); **high** = deserves its own issue. Judgment may override at the border.
- **Impact** — **high** if it hits any of: affects the CLI's end user, unblocks/accelerates future work, or reduces risk (fix, security, CI health, silent breakage); **low** otherwise or if purely cosmetic.

Each item lands in exactly one quadrant with its proposed route (per the glossary).

Done when every item has a quadrant and a proposed route.

### 3. Render the HTML report

Copy [`report.template.html`](./report.template.html), fill every `{{TOKEN}}`, write the result to the OS temp dir, and open it. Do **not** freehand HTML. See [HTML-REPORT.md](./HTML-REPORT.md) for tokens, card snippets, and **Chrome headless screenshot** steps (dark mode, card-based effort–impact matrix, colourful Mermaid pies).

After the HTML is written, take a PNG of the report with **Google Chrome headless** (from the Brewfile cask — see [HTML-REPORT.md](./HTML-REPORT.md): tall `--window-size` + `--virtual-time-budget` so Mermaid paints). Do **not** use `npx playwright`. Keep the PNG path for step 5.

Done when the report is written and opened, the PNG exists, and both paths are shown to the developer.

### 4. Confirmation gate — STOP

**Do not execute any side effect before this gate passes.** Present the whole batch — every item's quadrant and proposed route — via the **ask modal** (`AskQuestion`), a single approve/adjust prompt. The developer approves the whole set, or adjusts routes and then approves. Prefer a preview of the issue titles/bodies before creating anything.

**AFK (no human to answer the modal):** do **not** execute. Leave the report and proposed classification as a suggestion and stop here.

Done when the developer has explicitly approved the (possibly adjusted) routing for every item — never on assumption.

### 5. Execute the approved routing

Only after step 4. For each item, per its approved route:

- **⚡ Quick win / chosen 🍬 → Milestone refinement.** Aggregate into the milestone's Milestone refinement issue (create it once with the template below if absent; do not create an empty one). Born `needs-triage` + category (`enhancement`/`chore`).
- **🏆 Real win → Graduation** (if approved to defer): open its own tracer-bullet issue in the chosen milestone, born `needs-triage` + category (a large item is not AFK-ready from a one-line note). If approved to address now, it joins the current milestone's work.
- **🍬 not worth / 🕳️ Time sink → Discard.**
- **Unaddressed** leftover headings (present in the container but not scored/routed in this run) → treat like discard for body purposes (strike + 🕳️).

**Create every new issue as a child of the leftover container** using GitHub's `parentIssueId` (GraphQL `createIssue`). Also put `## Parent` → the leftover URL in the body (template below).

Record each disposition **in the container body**: routed/graduated items get a link to the new child issue; discarded or unaddressed items are struck through (`~~…~~`) with 🕳️ and left in place (so `/leftover` dedup can find them).

**Post exactly one decision-report comment** per triage run on the leftover container. That single comment must include, in order: AI disclaimer, routing summary (each item's quadrant, route, and child issue number or discard), the HTML report screenshot, and any close/process note that would otherwise become a separate comment.

Upload the PNG **before** posting — `gh image <png> --repo <owner/repo>` (extension `drogers0/gh-image`; may need `gh image extract-token` / `GH_SESSION_TOKEN`) — then post **one** `gh issue comment` with the markdown image embedded. Do **not** invent upload URLs or commit the PNG into the repo.

**Forbidden:** follow-up comments for the screenshot, a close note, or addenda. If something is missing after posting, **edit** that same comment (`gh api -X PATCH …/issues/comments/<id>`) — never post another for the same triage run.

Close the container with `gh issue close <n>` **without** `--comment` once every item is issue-ified or discarded/struck (any closing message belongs in the decision-report). **Never close the milestone** — at most suggest to the developer that it is ready to close.

Done when every item is routed, the body reflects each disposition, the single decision-report comment (with screenshot) is posted, and the container is closed (if fully destined).

```markdown

<link to the Leftover container>

## What to build

<end-to-end description of the aggregated wins (refinement) or the single change (graduation)>

## Acceptance criteria

```
